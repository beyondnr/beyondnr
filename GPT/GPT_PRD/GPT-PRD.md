## 1. 개요·목표

### 1.1 문제 정의 (현재 어떤 문제가 있나요?)

**현재 상황:**

* **수당 계산 실수 문제**

  - 인터뷰한 사장님 6명 중 67%가 수당 계산을 잘못한 적이 있음.
  - 월평균 3.2건씩 계산 오류가 발생하고, 이를 정정해야 함
  - 계산 오류 1건당 평균 120만원의 과태료나 임금 체불 위험이 있음
* **스케줄 관리가 복잡함**

  - 직원들이 언제 일할 수 있는지 알려주는 방법이 카카오톡, 전화, 수기로 분산되어 있음
  - 스케줄 승인까지 평균 48시간이 걸림
  - 결근이나 대타가 생겼을 때 공백이 생기는 비율이 18%
* **분쟁 대응이 어려움**

  - 인터뷰한 사장님 중 60%가 분쟁 발생 시 증거 자료가 없어서 어려움을 겪음
  - 최근 1년 내 합의금을 지급한 경험이 있는 사장님들이 많음
  - 직원들과의 신뢰도가 떨어짐

### 1.2 목표 (이 서비스로 해결하고 싶은 것)

**달성하고 싶은 결과:**

* **법규 준수 자동화**

  - 시스템이 자동으로 수당을 계산하고, 법규 위반이 생기기 전에 미리 경고
  - 계산 오류를 거의 없애고(0건 수준), 위반을 95% 이상 미리 막음
* **스케줄 관리 간소화**

  - 사장님이 승인만 하면 되는 간단한 3단계 가입 과정
  - 첫 스케줄을 만드는 데 10분 이내로 끝내는 비율을 70%까지 높임
  - 월말 급여 정산 시간을 5시간에서 30분으로 줄임 (90% 단축)
* **증빙 관리 자동화**

  - 모든 증거 자료를 자동으로 보관하고 필요할 때 바로 내보낼 수 있게 함
  - 분쟁 발생 시 증거 제출 성공률 100%
  - 사장님들의 불안감을 크게 줄임 (4.0점 → 1.5점)

## 2. 사용자와 페르소나

**페르소나란?** 실제 사용자들을 대표하는 가상의 인물입니다. 서비스를 설계할 때 "이 사람들이 어떻게 사용할까?"를 고려하기 위해 만듭니다.

### 주요 사용자 유형

1. **김영숙 (58세, 베이커리 운영)**

   - **고민**: 노동법에 따른 수당 계산이 너무 복잡해서 불안함 ("불안해서 잠이 안 옴")
   - **원하는 것**: 스케줄과 급여를 한 화면에서 동시에 볼 수 있는 기능
2. **박철민 (61세, 국밥집 운영)**

   - **고민**: 직접 스케줄을 만드는 것이 번거로움
   - **원하는 것**: 알바생이 직접 스케줄을 입력하고, 본인은 승인만 하면 되는 간단한 방식
   - **특징**: 복잡한 화면은 즉시 포기하는 성향
3. **이정희 (55세, 꽃집 운영)**

   - **고민**: 세무사로부터 법규 위반 경고를 받은 경험이 있음
   - **원하는 것**: 법규 위반을 미리 알려주는 "보험 같은" 자동 경고 시스템
   - **특징**: 법규 변경 사항에 매우 민감함
4. **윤지영 (57세, 의류 매장 운영)**

   - **고민**: 과거에 직원과 분쟁이 있었고, 증거 자료가 없어서 어려움을 겪음
   - **원하는 것**: 모든 변경 이력을 기록하고, 증빙 자료를 언제든지 내보낼 수 있는 기능
   - **특징**: 법정 분쟁에 대비한 신뢰할 수 있는 기록이 최우선

## 3. 사용자 스토리와 수용 기준 (AC)

**사용자 스토리란?** 사용자의 관점에서 "누가, 무엇을, 왜 원하는가"를 표현한 것입니다.
**수용 기준(AC)이란?** 이 기능이 제대로 작동했다고 인정받기 위한 구체적인 조건들입니다.

### Story 1 — 준법 급여 운영

**사용자 스토리:**

> "법규를 신경 쓰는 베이커리 사장으로서, 모든 수당이 자동으로 계산되기를 원합니다. 그래야 법에 맞게 급여를 지급할 수 있고 불안감이 없습니다."

**이 기능이 제대로 작동한다는 증거 (수용 기준):**

- **AC1: 수당 자동 계산 기능**

  - **상황**: 직원이 스마트폰으로 "이번 주에 언제 일할 수 있다"고 입력하고, 근무 시간이 쌓였을 때
  - **동작**: 사장님이 '미리보기' 버튼을 누름
  - **결과**:
    - 시스템이 주휴수당, 연장근로수당, 야간수당, 휴일수당을 자동으로 계산함
    - 계산 정확도는 99.5% 이상 (100번 중 99.5번은 정확함)
    - 계산 결과의 차이를 색상으로 표시해서 한눈에 알아보기 쉽게 함
- **AC2: 법규 위반 사전 차단 기능**

  - **상황**: 계산 결과가 주 52시간 초과, 휴게시간 부족, 연속 휴식 부족 등 법규 위반이 발견되었을 때
  - **동작**: 사장님이 승인 버튼을 누르려고 시도함
  - **결과**:
    - 승인 버튼이 비활성화되어 눌러지지 않음
    - 1초 이내에 어떤 규칙을 위반했는지 표시됨
    - 어떻게 수정해야 하는지 가이드가 함께 표시됨
- **AC3: 증빙 자료 자동 생성 및 보관 기능**

  - **상황**: 모든 법규 위반이 해결되어 정상적으로 승인했을 때
  - **동작**: 사장님이 PDF나 엑셀 파일로 증빙 자료를 요청함
  - **결과**:
    - 5초 이내에 다운로드 링크가 생성됨
    - 시스템이 자동으로 이 자료의 고유 번호(해시값)를 만들어서 1년간 보관함
    - 나중에 분쟁이 생겼을 때 이 자료가 변조되지 않았음을 증명할 수 있음
- **AC4: 입력 오류 자동 감지 및 알림 기능 (실패 케이스 대응)**

  - **상황**: 직원이 입력을 빠뜨리거나 중복으로 입력한 경우
  - **동작**: 시스템이 자동으로 모순을 감지함
  - **결과**:
    - SMS나 카카오톡으로 직원에게 알림을 보냄
    - 30분 이내에 다시 입력하도록 요청함
    - 승인 과정이 일시 중단됨 (입력이 정확해질 때까지)

### Story 2 — 1클릭 스케줄 승인

**사용자 스토리:**

> "기술에 익숙하지 않은 국밥집 사장으로서, 모아진 스케줄을 한 번의 클릭으로 승인하고 싶습니다. 그래야 운영이 덜 피로합니다."

**이 기능이 제대로 작동한다는 증거 (수용 기준):**

- **AC1: 승인 대기 화면 표시 기능**

  - **상황**: 최소 70% 이상의 직원이 스마트폰 링크를 통해 "언제 일할 수 있다"고 입력했을 때
  - **동작**: 사장님이 사장 전용 대시보드(메인 화면)를 열음
  - **결과**:
    - "승인 대기" 카드가 3단계로 표시됨 (검토 → 조정 → 공지)
    - 각 단계가 2초 이내에 화면에 나타남 (로딩이 빠름)
- **AC2: 스케줄 변경 및 알림 기능**

  - **상황**: 사장님이 스케줄 화면에서 드래그&드롭으로 교대 시간을 변경했을 때
  - **동작**: 저장 버튼을 누름
  - **결과**:
    - 시스템이 임시 버전을 만들어서 저장함
    - 15초 이내에 변경된 직원들에게 카카오톡이나 SMS로 알림이 발송됨
    - 모든 알림이 한 가지 채널로 통일되어 관리가 쉬움
- **AC3: 확인 여부 자동 추적 기능**

  - **상황**: 승인된 스케줄이 확정된 후
  - **동작**: 직원이 스케줄 확인을 하지 않은 채로 12시간 이상 지남
  - **결과**:
    - 시스템이 자동으로 직원에게 다시 알림을 보냄
    - 확인하지 않은 직원 비율이 10%를 넘으면 사장님에게도 텍스트 알림을 보냄

## 4. 기능 요구사항 (Functional, MSCW)

**기능 요구사항이란?** 서비스에 포함되어야 할 기능들의 목록입니다.**MSCW란?** 기능의 우선순위를 나타내는 분류입니다.

- **Must**: 반드시 있어야 하는 기능 (없으면 서비스가 의미 없음)
- **Should**: 있으면 좋은 기능 (없어도 서비스는 작동하지만 사용자 경험이 떨어짐)
- **Could**: 나중에 추가해도 되는 기능 (있으면 좋지만 우선순위는 낮음)
- **Won't**: 이번 버전에는 포함하지 않는 기능 (나중 버전에서 고려)

| -------------- | ---------------- | ------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------- | -------------------------------------------- |
| **F-01** | **Must**   | **3단계 온보딩 마법사** `<br>`(가입 → 매장 설정 → 첫 스케줄 만들기)               | 70%의 사용자가 10분 이내에 완료 `<br>`Excel 템플릿 대비 4.5배 빠름   | SMS 인증 기능, 매장 정보 템플릿 데이터베이스 |
| **F-02** | **Must**   | **직원용 모바일 가용시간 제출 폼** `<br>`(QR코드나 링크로 접근)                     | 직원 참여율 80% 이상 `<br>`로그인 없이도 제출 가능                   | 짧은 링크 생성 기능, 간편 본인확인 기능      |
| **F-03** | **Must**   | **사장님 1클릭 승인 및 공지 기능**                                                    | 승인 시간 48시간 → 6시간으로 단축 `<br>`Excel 대비 8배 빠름         | 알림 발송 서비스                             |

| **F-06** | **Should** | **변경 이력 기록 및 서명 추적** `<br>`(모든 변경 사항을 기록하고 변조 방지)         | 모든 사건별 로그 100% 기록 `<br>`SHA256 방식으로 서명                | 파일 저장소 (Object storage)                 |
| **F-07** | **Should** | **간단한 리포트 및 인쇄용 화면** `<br>`(오프라인으로 출력 가능)                     | 리포트 생성 요청 후 30초 이내 생성 `<br>`매장 벽에 붙일 수 있는 형태 | 인쇄에 적합한 화면 디자인 (CSS)              |
| **F-08** | **Could**  | **출퇴근 기록 기본 연동** `<br>`(수기 입력 또는 사진 업로드)                        | 급여 계산 정확도 ±2% 이내 유지                                        | 사진 인식 기능 (OCR), 사진 업로드 기능       |
| **F-09** | **Won't**  | **POS 시스템/급여 솔루션과의 연동** `<br>`(이번 버전에는 포함 안 함)                | 나중 버전에서 고려                                                     | 외부 업체와의 계약 필요                      |

**용어 설명:**

- **온보딩**: 처음 서비스를 사용하기 위해 가입하고 설정하는 과정
- **가용시간**: 직원이 일할 수 있는 시간
- **규칙 엔진**: 법규를 자동으로 체크하는 시스템
- **증빙**: 법적 분쟁 시 제출하는 증거 자료
- **Audit Log**: 모든 변경 사항을 기록한 로그
- **SHA256**: 데이터가 변조되지 않았음을 증명하는 암호화 방식
- **OCR**: 사진 속 글자를 읽어서 텍스트로 변환하는 기술
- **POS**: 매장에서 결제를 처리하는 시스템

## 5. 비기능 요구사항 (NFR)

**비기능 요구사항이란?** 기능이 아니라 서비스의 품질, 성능, 보안 등에 대한 요구사항입니다. "어떻게 작동해야 하는가"를 정의합니다.

### 5.1 성능 (서비스가 얼마나 빠른가?)

- **스케줄 보드 조회**: 95%의 경우 0.8초(800ms) 이내에 화면이 나타나야 함
- **승인 처리**: 95%의 경우 1.2초 이내에 승인이 완료되어야 함
- **PDF 생성**: 직원 50명 기준으로 5초 이내에 PDF 파일이 생성되어야 함

### 5.2 신뢰성 (서비스가 얼마나 안정적인가?)

- **서비스 가용성**: 한 달 중 99.5% 이상의 시간 동안 서비스가 정상 작동해야 함 (한 달에 약 3.6시간만 중단 가능)
- **규칙 엔진 오류율**: 법규 체크 시스템의 오류율은 0.2% 이하 (오류 발생 시 자동으로 재시도)
- **알림 전송 성공률**: SMS/카카오톡 알림이 97% 이상 성공적으로 발송되어야 함

### 5.3 보안 및 비용 (데이터 보호와 운영 비용)

- **데이터 암호화**:
  - 모든 직원 데이터는 AES-256 방식으로 암호화하여 저장
  - 인터넷 통신은 TLS 1.3 방식으로 암호화 (최신 보안 표준)
- **데이터 저장 위치**: 국내 리전(서버 위치)만 사용 (해외 서버 사용 금지)
- **개인정보 최소 수집**: GDPR(유럽 개인정보보호법) 및 근로기준법 기준에 따라 최소한만 수집 (직원 1명당 8개 항목 이하)
- **운영 비용**: 활성 매장 1곳당 월 운영 비용 3,000원 이하 유지

### 5.4 모니터링 (서비스 상태를 어떻게 확인하는가?)

- **실시간 모니터링**:
  - Datadog 대시보드에서 API 응답 시간, 규칙 엔진 실패, 알림 대기열 상태를 실시간으로 확인
  - 대시보드 이름: `prd-lawfulshift-core`
- **자동 알림**:
  - API 응답 시간이 1.2초를 넘는 경우가 5분 이상 지속되면 심야에도 자동으로 알림 발송
  - PagerDuty 시스템을 통해 담당자에게 알림

**용어 설명:**

- **p95**: 95%의 경우를 의미 (예: p95 800ms = 100번 중 95번은 0.8초 이내)
- **가용성**: 서비스가 정상 작동하는 시간 비율
- **AES-256**: 강력한 데이터 암호화 방식
- **TLS 1.3**: 인터넷 통신을 암호화하는 최신 보안 프로토콜
- **리전**: 서버가 위치한 지역
- **GDPR**: 유럽연합의 개인정보보호 규정
- **Datadog**: 서비스 모니터링 도구
- **PagerDuty**: 장애 발생 시 담당자에게 알림을 보내는 시스템

## 6. 데이터·인터페이스 개요

**이 섹션이란?** 서비스에서 다루는 주요 데이터와 시스템 간 통신 방법을 설명합니다. AI에게 개발을 요청할 때 이 정보가 필요합니다.

### 6.1 핵심 데이터 구조 (서비스에서 관리하는 주요 정보)

서비스에서 다루는 주요 데이터들의 구조입니다:

1. **매장 정보 (Store)**

   - 매장 고유 번호, 업종, 영업시간, 대리인 정보
2. **직원 정보 (Employee)**

   - 직원 고유 번호, 역할(사장/직원), 계약 유형(정규/비정규), 시급, 연락처
3. **가용시간 제출 정보 (AvailabilitySubmission)**

   - 제출 고유 번호, 직원 번호, 일할 수 있는 시간대, 제출 시간, 제출 방법(카톡/전화/앱)
4. **스케줄 정보 (Schedule)**

   - 스케줄 고유 번호, 매장 번호, 주차, 확정 여부, 승인한 사람, 버전
5. **스케줄 템플릿 (ShiftTemplate)**

   - 템플릿 고유 번호, 요일, 기본 적용 규칙 번호
6. **법규 규칙 정보 (ComplianceRule)**

   - 규칙 고유 번호, 법 조항, 계산 공식, 발효일, 위반 시 심각도
7. **변경 이력 로그 (AuditLog)**

   - 로그 고유 번호, 변경된 항목, 변경 행위, 변경 전 값, 변경 후 값, 변조 방지 서명

### 6.2 시스템 간 통신 방법 (API / 인터페이스)

**API란?** 다른 시스템이나 앱이 우리 서비스와 소통하는 방법입니다. AI에게 개발을 요청할 때 이 형식을 참고하면 됩니다.

1. **직원 가용시간 제출 API**

   - **주소**: `POST /api/v1/availability`
   - **접근**: 누구나 접근 가능 (public)
   - **제한**: 1분에 30번까지만 요청 가능
   - **입력 정보**: 직원 토큰, 주차, 일할 수 있는 시간대 목록
   - **오류 처리**: 입력이 잘못되면 400 오류 코드와 원인을 반환
2. **스케줄 조회 API**

   - **주소**: `GET /api/v1/schedules/{id}` (id는 스케줄 번호)
   - **반환 정보**: 교대 시간, 법규 위반 목록, 승인 상태
   - **성능**: 30초간 캐싱 (같은 요청은 30초간 빠르게 응답)
3. **스케줄 승인 API**

   - **주소**: `POST /api/v1/schedules/{id}/approve`
   - **입력 정보**: 조정 사항 목록, 알림 채널
   - **오류 처리**: 법규 위반이 있으면 409 오류 코드와 위반 규칙 번호를 반환
4. **급여 리포트 조회 API**

   - **주소**: `GET /api/v1/reports/payroll?weekOf=YYYY-WW` (YYYY-WW는 연도-주차)
   - **반환 정보**: 수당 목록, PDF 다운로드 링크, 증빙 자료 고유 번호
5. **법규 위반 알림 (Webhook)**

   - **주소**: `/events/violation`
   - **용도**: 외부 파트너(노무사)에게 법규 위반 발생 시 자동으로 알림
   - **보안**: 서명 헤더 포함 (위조 방지)
6. **관리자용 법규 규칙 업데이트**

   - **형식**: CSV 파일로 가져오기/내보내기
   - **접근 권한**: 노무사만 접근 가능
   - **용도**: 법규 규칙 테이블 업데이트

**용어 설명:**

- **API**: 다른 시스템과 소통하는 방법
- **Webhook**: 특정 사건이 발생했을 때 자동으로 알림을 보내는 기능
- **캐싱**: 자주 요청하는 데이터를 임시 저장해서 빠르게 응답하는 방법
- **CSV**: 엑셀에서 열 수 있는 표 형식의 파일

## 7. 범위, 리스크·가정·의존성

**이 섹션이란?** 이번 버전에 포함할 것과 포함하지 않을 것, 그리고 발생할 수 있는 문제와 대응 방법을 정리한 것입니다.

### 7.1 포함 범위 (이번 버전에 만들 기능)

**MVP v1에 포함되는 기능:**

- 3단계 온보딩 (가입 → 매장 설정 → 첫 스케줄)
- 직원 가용시간 제출 (링크/QR코드로 접근)
- 사장님 1클릭 승인 기능
- 법규 준수 규칙 엔진 (주휴수당, 연장근로수당, 야간수당, 휴일수당, 주 52시간, 휴게시간, 연속 휴식 체크)
- 증빙 자료 생성 (PDF/엑셀 파일)
- 변경 이력 기록 (Audit Log v1)
- 알림 기능 (카카오톡/SMS 중 하나만 사용)

### 7.2 제외 범위 (이번 버전에는 만들지 않는 기능)

**나중 버전에서 고려할 기능:**

- 여러 지점 관리 기능
- 역할별 권한 관리 (사장/매니저/직원 등)
- POS 시스템과의 연동
- 급여 솔루션과의 API 연동
- 다국어 지원 (영어, 중국어 등)
- AI를 활용한 스케줄 자동 추천
- 여러 알림 채널 동시 사용 (이메일, 푸시 알림 등)

### 7.3 리스크 및 대응 방법 (발생할 수 있는 문제와 해결책)

**R1: 사용자가 디지털 기기에 익숙하지 않을 수 있음**

- **대응 방법**:
  - 대리인이 대신 사용할 수 있는 모드 제공
  - 전화 상담 서비스 제공
  - 오프라인으로 출력할 수 있는 기능 제공
  - 온보딩 가이드 영상 제작

**R2: 법령이 변경되었을 때 반영이 늦을 수 있음**

- **대응 방법**:
  - 노무사와 자문 계약 (2주 이내 반영 보장)
  - 법규 규칙 테이블을 빠르게 업데이트할 수 있는 시스템 구축
  - 업데이트 시 체크리스트 준수

**R3: 개인정보 유출로 인한 신뢰도 하락**

- **대응 방법**:
  - 필드별 암호화 (각 항목별로 암호화)
  - 접근 통제 강화 (MFA: 이중 인증)
  - 분기별로 보안 침해 대응 훈련 실시

**R4: 알림 채널(카카오톡/SMS)이 실패할 수 있음**

- **대응 방법**:
  - SMS 백업 라우팅 (카카오톡 실패 시 SMS로 자동 전환)
  - 알림 발송 성공률을 모니터링하고, 95% 미만이면 자동으로 다른 채널로 전환

### 7.4 가정 및 의존성 (이것들이 가능하다고 가정하는 것들)

**외부에서 제공되어야 하는 것들:**

- 고용노동부에서 법규 규칙 데이터를 공개 API나 정기 공지로 제공 (월 1회 업데이트 예상)
- 카카오톡/SMS 게이트웨이 서비스가 99.9% 이상 가동률 유지, 장애 시 4시간 이내 복구
- 노무사 파트너가 법규 변경 시 검토 메모를 48시간 이내에 제공 (ADR-2025-02 문서 참조)

**용어 설명:**

- **MVP**: Minimum Viable Product의 약자로, 최소한의 기능만 포함한 첫 버전
- **SLA**: Service Level Agreement의 약자로, 서비스 수준 협약 (예: 99.9% 가동률)
- **MFA**: Multi-Factor Authentication의 약자로, 이중 인증
- **PII**: Personally Identifiable Information의 약자로, 개인을 식별할 수 있는 정보
- **핫스왑**: 서비스를 중단하지 않고 빠르게 업데이트하는 방법

## 8. 실험·롤아웃·측정

**이 섹션이란?** 서비스를 실제 사용자에게 제공하기 전에 테스트하고, 점진적으로 확대하는 계획입니다.

### 8.1 베타 테스트 계획

**대상**: 4분기(Q4)에 디지털 기기에 익숙하지 않은 사업장 30곳 (베이커리, 국밥집, 꽃집)
**기간**: 6주간 베타 테스트
**롤아웃 단계** (점진적 확대):

1. **내부 dogfood**: 우리 팀이 먼저 사용해보기
2. **컨시어지 온보딩**: 상담원이 직접 도와주며 사용
3. **Self-serve**: 사용자가 스스로 사용

### 8.2 실험 가설 및 측정

**실험 가설:**

> "3단계 온보딩 + 1클릭 승인" 기능이 월말 정산 시간을 90% 단축하고, 법규 위반 경고 처리율을 80% 달성할 것이다.

**실험 설계:**

- **방법**: A/B 테스트 (두 가지 방법 비교)
  - A그룹: 컨시어지(상담원)가 도와주는 방식
  - B그룹: 사용자가 스스로 하는 방식
- **대상**: 총 60개 매장 (각 그룹당 30개)
- **측정 지표**: 온보딩 시간, 승인 리드타임, 오류율

**성공 기준:**

- 통계적으로 유의미한 차이가 있어야 함 (p-value < 0.05)
- 목표 수치를 달성해야 함
- 이탈률(churn)이 10% 미만이어야 함

### 8.3 경쟁사 대비 벤치마크

**비교 대상**: Excel/수기 방식, 기존 SaaS 서비스, LawfulShift
**측정 방법**: 같은 주차 데이터를 세 가지 방법으로 처리하고 비교
**목표 성과**:

- 계산 정확도: Excel/수기 대비 1.5배 향상
- 승인 속도: Excel/수기 대비 8배 향상
- 운영 비용: Excel/수기 대비 20% 절감

### 8.4 롤아웃 가드레일 (안전장치)

**롤아웃을 중단해야 하는 경우:**

- 법규 위반 경고 대응률이 60% 미만일 때
- PDF 생성 실패율이 2%를 넘을 때

**중단 시 조치**: 롤아웃을 즉시 중지하고 긴급 수정(핫픽스) 후 재개

**용어 설명:**

- **베타**: 정식 출시 전 테스트 단계
- **롤아웃**: 서비스를 점진적으로 확대하는 과정
- **Dogfood**: 우리가 만든 서비스를 우리가 먼저 사용해보는 것
- **A/B 테스트**: 두 가지 방법을 비교해서 더 나은 것을 찾는 실험
- **p-value**: 통계적으로 의미 있는 차이가 있는지 판단하는 수치
- **Churn**: 서비스를 그만두는 비율 (이탈률)
- **벤치마크**: 경쟁사와 비교하는 기준
- **핫픽스**: 긴급하게 문제를 수정하는 것
